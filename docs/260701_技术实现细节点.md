## 一、RCA Agent 置信度量化

### 1.1 公式

```
confidence = evidence_strength × coverage
若命中链路 ≥ 2：× 0.85
HITL 阈值：0.7
```

### 1.2 evidence_strength（证据强度）

FMEA 规则引擎扫描每条从 MCP 返回的证据，按来源标记强度：

| 证据来源 | 强度 | 说明 |
|---------|:----:|------|
| MES 批次追溯（结构化工序数据） | 0.9 | 确定性最高，有明确字段 |
| LIMS 检测报告 | 0.85 | 实验室签字数据，可追溯 |
| SCADA 时序异常窗口 | 0.8 | 有明确时间戳和阈值 |
| ERP 来料批次记录 | 0.7 | 供应商报告，确定性中等 |
| 纯文本描述（人工备注） | 0.3 | 非结构化，凭经验 |

证据强度 = 有效证据的加权平均分 / **本缺陷 FMEA 树涉及的数据源数**

> **⚠️ 注意分母**：分母是 FMEA 树中与本缺陷相关的数据源数量（静态），不是实际查到的数据源数（动态），也不是所有 MCP Server 数。否则一个只查了 2 个系统的缺陷会因为"没查另外 3 个"而被拉低置信度。

### 1.3 coverage（FMEA 覆盖率）

FMEA 因果树中命中的路径比例：

```
coverage = 完整命中的因果路径数 / FMEA 树中该缺陷的总路径数
```

### 1.4 设计要点

- **确定性替代 LLM 自评**：同一个输入永远输出同一个置信度，可复现、可审计
- **只读计算**：不写数据库，不产生副作用
- **覆盖率不足时降级**：FMEA 无命中 → Reflector 走 DEGRADE 模式（LLM 兜底 + 低置信度标记 + HITL）

### 1.5 边界情况

| 场景 | 行为 |
|------|------|
| FMEA CSV 为空 | DEGRADE → 纯 LLM + 置信度 0.3 + 强制 HITL |
| 所有 MCP 全部超时 | 0 证据 → confidence=0 → HITL |
| Planner 报错 | Orchestrator `_fallback_rca()` 兜底，置信度固定 0.3 |
| Executor 部分失败 | 部分证据集参与计算，不强求全部获取 |

## 二、HITL 挂起与恢复（LangGraph + A2A）

### 2.1 整条链路

```
RCA Agent 内部
  Reflector(confidence < 0.7)
    → interrupt()
    → GraphInterrupt 异常

AsyncA2AServer.handle_task()
    → _is_graph_interrupt() 检测异常类型名
    → _interrupted_task()
    → 构造 INPUT_REQUIRED Task(含 thread_id)

A2A 协议层
    → HTTP JSON-RPC 返回 task_status=INPUT_REQUIRED

Orchestrator
    → 收到 INPUT_REQUIRED
    → 把 thread_id 写进 PlatformContext.rca.thread_id
    → 返回 Gateway → 工程师

工程师签核
    → POST /tasks/resume {thread_id, feedback}

AsyncA2AServer.handle_resume()
    → aget_state(config) 验证中断确实 pending
    → Command(resume=feedback)
    → graph.ainvoke(resume_input, config)
    → 从 interrupt() 后继续执行 → Reporter → COMPLETED
```

### 2.2 LangGraph 底层的三个关键函数

| 函数 | 作用 |
|------|------|
| `interrupt(value)` | 挂起当前图执行，保存 checkpoint，抛异常 |
| `Command(resume=value)` | resume 时注入用户反馈，从断点后继续 |
| `aget_state(config)` | 检查 thread_id 对应的 checkpoint 是否有 pending interrupt |

### 2.3 三层状态机映射

| 层 | 状态机 | 生命周期 | 存储 |
|----|--------|---------|------|
| **A2A TaskState** | SUBMITTED → RUNNING → INPUT_REQUIRED → RUNNING → COMPLETED | 跨 HTTP 请求 | 无状态（每次请求携带） |
| **LangGraph GraphState** | Planner → Executor → Reflector → interrupt() → Reflector(resume) → Reporter | 单进程内 | Checkpointer (MemorySaver) |
| **PlatformContext** | running → hitl → done | 跨多个 Agent 服务 | Redis/PG |

**三者的桥接**：`_thread_map[session_id] = f"lg_{session_id}_{uuid}"`

### 2.4 安全边界

- **重复 resume**：`aget_state` 检查 `state.interrupts` 是否 pending，没有则拒绝
- **多轮 HITL**：resume 后又遇到 interrupt → 返回新的 INPUT_REQUIRED
- **超时**：当前无 HITL 超时取消逻辑（已知限制）
- **thread_id 碰撞**：`uuid.hex[:8]` 后缀降低了 collision 概率

## 三、三层记忆架构

### 3.1 分层设计

```
STM ─── Redis ─── 单次 A2A Task 上下文
  短期，TTL=24h，按 session_id key-value

Working ─── PostgreSQL ─── 未结案工单、工艺记录
  业务持久，跨 session

LTM ─── Neo4j + Milvus ─── FMEA 因果图 + Golden Case 向量 + SOP
  长期知识，独立 ETL 管道维护
```

### 3.2 Working Memory（PostgreSQL 持久化）

#### 3.2.1 设计

Working Memory 层承担「跨 session 未结案工单持久化」。PlatformContext 序列化为 JSONB 存入 `working_memory` 表：

```
表 working_memory
  session_id  TEXT PRIMARY KEY   — A2A session 标识
  tenant_id   TEXT               — 租户/工厂隔离
  batch_id    TEXT               — 批次号，便于按批次检索
  task_status TEXT               — running / hitl / completed / failed
  context     JSONB              — 完整 PlatformContext
  created_at  TIMESTAMPTZ        — 首次创建
  updated_at  TIMESTAMPTZ        — 最后更新
```

- **未结案工单** = `task_status NOT IN ('completed', 'failed', 'cancelled')`
- **TTL 清理**：已完结工单超过 7 天后自动清理（`cleanup_expired()`），不删除进行中的工单
- **自动建表**：`PostgresSessionStore._ensure_table()` 首次启动幂等建表，不需手动跑 migration
- **连接池**：`asyncpg.create_pool(min_size=1, max_size=4)`，不阻塞 app 启动

#### 3.2.2 与 SessionStore 接口对齐

```python
class SessionStore(Protocol):
    async def get(session_id) -> PlatformContext | None
    async def save(ctx) -> None
    async def exists(session_id) -> bool
    async def list_active(tenant_id) -> list[PlatformContext]  # ← 新增
```

`list_active()` 是 Working Memory 独有的能力——Redis 和 Memory 后端不擅长全量扫描，标记为未实现或仅支持小规模。

#### 3.2.3 配置切换

```env
CONTEXT_BACKEND=postgres        # memory | redis | postgres
POSTGRES_DSN=postgresql://battery:battery@localhost:5432/battery_agent
```

三个后端实现同一接口，运行时一键切换。生产推荐 postgres（持久 + 可查询），开发用 memory（零依赖）。

### 3.3 PlatformContext（跨 Agent 黑板）

```python
class PlatformContext(BaseModel):
    session_id: str           # Redis key
    trace_id: str             # 全链路审计
    batch_id: str | None
    defect_type: str | None
    prior_evidence: list
    rca: RcaContext           # thread_id, root_cause, confidence
    report_8d: Report8dContext
    artifacts: dict[str, Any] # ← 通用扩展槽，新增 Agent 不改 Schema
    task_status: str
    current_step: str
```

**存储方式**：`RedisSessionStore` — `setex("platform:context:{session_id}", TTL, JSON)`

### 3.3 新增子系统

新子系统直接写 `ctx.artifacts["equipment-health-agent"] = {...}`。下游按 agent_name 读取。

**已知限制**：
- `artifacts` 无 JSON Schema 校验（写错字段名静默丢失）
- 读取方需要知道写入方的字段结构（隐含契约）
- 所有数据在一个 Redis key 内，大 blob 会撑大序列化体积

**演进方向**：artifact_refs 分离（context 只存元数据 + blob_id，大块数据独立存储）

## 四、混合检索（Milvus + Neo4j RRF 融合）

### 4.1 通道 A：Milvus 语义相似

```
症状描述 → text-embedding-3-small → 768d vector → Milvus cosine search → top-K
```

即使 defect_type 不在历史库里，语义相似的案例也能召回。

### 4.2 通道 B：Neo4j 图结构

```
defect_type → MATCH (d:Defect)-[:CAUSED_BY*1..4]->(rc:RootCause)
           → MATCH (gc:GoldenCase)-[:HAS_DEFECT]->(d2)-[:CAUSED_BY*1..4]->(rc)
           → 共享根因的案例
```

即使症状描述不同，但因果路径相同也能召回。

### 4.3 RRF 融合

```python
k = 60
score = 1/(k + rank_milvus) + 1/(k + rank_neo4j)
```

两个通道都命中的案例排最前。

### 4.4 数据模型

- **Golden Case**（15 条，12 种缺陷，9 道工序）
- **Neo4j 节点**：`Defect` / `RootCause{category: equipment/material/process_parameter/environment/operation}` / `GoldenCase`
- **Neo4j 关系**：`[:HAS_DEFECT]` / `[:CAUSED_BY{confidence}]`
- **Milvus 索引**：`golden_cases` collection，COSINE 度量，IVF_FLAT 索引

### 4.5 ETL

```bash
python scripts/etl_golden_cases.py --rebuild
```

CSV 是 Source of Truth → embedding → Milvus 向量 + Neo4j 节点关系同步。

## 五、Reporter SubAgent 设计

### 5.1 按搜索歧义划分粒度

| 版本 | 子 Agent 数 | LLM 调用 | 延迟 | 问题 |
|:----:|:----------:|:--------:|:----:|------|
| v1 | 3（D4+D5+D6） | 25 次 | 94s | D4/D6 零歧义却消耗 LLM |
| v2 | tool 函数 | 死循环 | ∞ | tool 返回字符串，LLM 不知道完成 |
| **v3** | **1（仅 D5）** | **9 次** | **42s** | **D4/D6 预格式化注入，不经过 LLM** |

### 5.2 D4/D6 预格式化机制

```python
# deep_agent_runner.py
d4_block = f"## D4 根因\n\n{req.root_cause}\n"
# 用 [D4_PREFORMATTED] 标签注入 user message
# 主 Agent 收到时已是成品 Markdown
user_message = f"...[D4_PREFORMATTED]\n{d4_block}\n[D6_PREFORMATTED]\n{d6_block}\n..."
```

**三层防御**：
1. Python 字符串拼接，不经过 LLM
2. system prompt 标注「原样使用，禁止改写」
3. 离线 `rule_judge_reporter_d4_locked()` 断言检查

### 5.3 降级路径

```python
# 四种情况全部降级到 template（无 LLM 调用）
if deepagents not installed → template
if reporter_mode != "deep_agent" → template
if llm_api_key not configured → template
if agent.ainvoke() throws → template
```

### 5.4 子 Agent 的审计

子 Agent `d5_capa_planner` 的内部 tool 调用（`search_sop`、`search_golden_case`）是直接 Python 函数调用，不经 `ToolRegistry`。但加几行 `logger.info(..., trace_id=get_trace_id())` 即可补审计，不需拆分服务。读公共知识库不需要 RBAC 校验。

## 六、Playbook DSL 引擎

### 6.1 设计目标

替代 Orchestrator 中 211 行的硬编码 if-else（`_run_playbook()`）。

### 6.2 引擎架构

```
config/playbooks.yaml → PlaybookEngine.load()
                        → steps[].condition 检查
                        → steps[].agent → call_step(agent, step_def)
                        → steps[].hitl_check → 自动 HITL
                        → steps[].context_write → 自动映射
                        → steps[].fallback → 自动兜底
```

### 6.3 条件表达式（安全解析，不用 eval）

```yaml
condition: "not skip_triage and not defect_type"
condition: "batch_id"
condition: "rca.root_cause"
```

引擎自己解析 token 序列，只支持 `not` / `and` / `or` / 字段名 / 点路径，不支持任意 Python 表达式。

### 6.4 新增剧本流程

```
1. 在 config/playbooks.yaml 加新条目
2. 保证目标 Agent 有 AgentCard
3. 重启 orchestrator（或用 Redis 信号触发 reload）
```

## 七、熔断器与高可用

### 7.1 CircuitBreaker 状态机

经典 HALF_OPEN + 固定冷却不适合 MCP 内部调用——请求量低、冲击可控。改用**概率放行**：

```
失败3次                       探针成功
CLOSED ──────→ OPEN(5%探针率) ──────→ CLOSED
  ↑                                     │
  └───────────── 探针失败 ──────────────┘
```

OPEN 状态下 95% 请求直接拒绝（fast-fail），5% 作为探针放行：
- **探针成功** → 立即转 CLOSED，秒级恢复，不等固定定时器
- **探针失败** → 继续 OPEN，探针失败不计入 threshold（避免累计误解）

vs 经典模式对比：

| 维度 | 经典（HALF_OPEN + 30s） | 本方案（概率放行） |
|------|------------------------|------------------|
| 恢复速度 | 平均等待 ≥30s | 平均 1/probe_rate 次请求（≈20 次） |
| 状态数 | 3（CLOSED/OPEN/HALF_OPEN） | **2（CLOSED/OPEN）** |
| 对下游冲击 | 1 个探针 | 5% 请求，MCP 场景下可忽略 |
| 参数 | cooldown_seconds + half_open_max | 仅 probe_rate |

### 7.2 粒度

当前：**MCP Server 维度**（一个 MES 熔断不影响 SCADA）。
更优：**Tool 维度**（`mes.query_batch_trace` 熔断不影响 `mes.get_process_params`），待优化。

### 7.3 并发安全

asyncio 单线程，无 race condition。

### 7.4 横向扩展问题

多实例部署时熔断状态**不共享**，探针独立。生产化需改为 Redis 存储熔断状态。

### 7.5 三条降级路径

| 故障 | 降级 | 效果 |
|------|------|------|
| RCA Agent 超时 / MCP 证据不足 | `_fallback_rca()` 构造初判根因 | 置信度 0.3，强制 HITL |
| Reporter Deep Agent 异常 | 切 template 模式 | 确定性格式化，0 LLM |
| Planner LLM 不可用 | 切 rule 引擎 | 正则+关键词匹配 |

## 八、MCP 数据同步与数据源

### 8.1 当前状态（stub）

```python
@mcp.tool()
async def get_process_params(...):
    return {"coating_thickness_mean_um": 95.2}  # 硬编码 mock 数据
```

所有 MCP Server 当前返回固定 mock 数据，没有连接真实产线数据库。

### 8.2 真实接入后

```python
@mcp.tool()
async def get_process_params(batch_id, ...):
    row = await db.fetchrow("SELECT * FROM params WHERE batch_id=$1", batch_id)
    return row.to_dict()
```

每次 tool 调用直连数据库，**无缓存层**。MCP 协议本身不关心数据来源，切真实数据不改 Agent 代码。

### 8.3 数据新鲜度

- 分析场景（批次 RCA）分钟级延迟完全可接受
- SCADA 异常检测在 MCP Server 侧做（靠近数据源），只返回结果给 Agent
- 控制场景（PLC 改参）走 Safety Agent 独占门闩

## 九、LLM 调用策略

### 9.1 RCA Agent 内部

| 节点 | 调 LLM？ | 原因 |
|------|:--------:|------|
| Planner | ✅ | NL → 结构化取证计划，需要语义理解 |
| Executor | ❌ | 纯 MCP 网络调用，Worker 逻辑 |
| Reflector | ⚠️ FMEA 不命中时才调 | FMEA 命中 → 规则判定（0 token）；不命中 → LLM 兜底 |
| HITL | ❌ | 流程中断点，无计算 |
| Reporter | ✅ | 生成 8D 草稿，需要 NL 生成 |

### 9.2 全链路总调用数

```
triage:    0（规则引擎）
trace:     0（纯 MCP）
RCA:       2-3（Planner + 可选 Reflector + Reporter）
Reporter:  9（Deep Agent，含子 Agent）
总计:      ~11-12 次 LLM 调用
```

### 9.3 模型路由

按 `DataSensitivity(LOW/MEDIUM/HIGH) × TaskDifficulty(TRIVIAL→COMPLEX)`：

| 路由维度 | 决策 | 示例 |
|---------|------|------|
| sensitivity=LOW | 送 external API | 缺陷描述、8D 报告 |
| sensitivity=MEDIUM | 优先 local vLLM | 工艺参数摘要 |
| sensitivity=HIGH | 抛异常，不降级 | 当前无节点标记 HIGH |

## 十、Triage Agent — 意图识别与异常分诊

### 10.1 设计

Triage Agent 是用户与品质域推理之间的**第一层 NLU**，替代 Planner 中粗糙的关键词正则分诊。

```
用户: "这批电池B202406001容量偏低，之前也有过类似情况"

Triage Agent:
  → 多意图拆解: ["容量偏低", "之前也有过类似情况"]
  → 实体抽取:   batch_id="B202406001"
  → defect_type: "capacity_low"
  → severity:    "high"  (历史复现→升级)
  → suggest:     "trace" (有 batch_id)
  
输出到 PlatformContext:
  defect_type, severity, triage_result
```

### 10.2 双模架构

| 模式 | 触发条件 | 能力 | 延迟 |
|------|---------|------|:----:|
| **LLM** | API key 已配置 | 语义理解、多意图拆解、上下文推理 | ~1-2s |
| **Rule** | 无 API key / LLM 超时 / 低置信度 | 16 类关键词正则 + 3 级严重度 + 批次号提取 | ~1ms |

LLM 返回 `confidence < 0.5` 时自动降级到规则——防御性设计，不用用户等一个不靠谱的 LLM 回答。

### 10.3 16 类缺陷映射

覆盖匀浆→装配全工序已知缺陷类型，未知缺陷默认 `unknown_defect` + `severity=low` + 强制 HITL。

### 10.4 与分诊 stub 的兼容

- Orchestrator **优先 A2A 调 triage-agent**
- 失败或超时 → 降级到 `triage_stub.resolve_triage()`（原来的 4 条关键词正则）
- 两种输出都是 `TriageResponse`，下游无感知
