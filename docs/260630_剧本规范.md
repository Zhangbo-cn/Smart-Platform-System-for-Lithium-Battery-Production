# Playbook 剧本规范

> **版本**：v1.0 | 2026-06-29  
> **权威来源**：本文档为平台 **6 个 Playbook** 的唯一清单；`PLATFORM_SPEC` / `AGENT_CATALOG` / `ARCHITECTURE_DIAGRAMS` 均引用本文。  
> **执行主体**：**Client Gateway** 接请求 → **Planner**（可选，Agent）→ **Playbook Orchestrator** 按剧本 `tasks/delegate` 调度业务能力服务；**业务能力服务不互连**。

---

## 1. 执行模型

```
调用方 → Client Gateway → Planner? → Playbook Orchestrator → 业务能力服务（A2A）
                              ↘ PlatformContext（Session Store）↗
```

| 角色 | 职责 |
|------|------|
| **Client Gateway** | 统一入口；`POST /v1/assistant/tasks`；SSE 推送 `task_status` |
| **Planner** | Agent：开放目标 ReAct；产出 `playbook` + 参数交 Orchestrator |
| **Playbook Orchestrator** | 按 `playbook` + capability 寻址；维护父子 `task_id`；写 PlatformContext |
| **业务能力服务** | 单域能力；仅响应 Orchestrator 委派（含 Agent 与 Worker） |

**原则**：

- 所有跨服务步骤经 **Playbook Orchestrator**，禁止业务能力服务直连。
- RCA 需 **显式** 进入（`task_type=rca` / 用户确认 / 白名单集成），见各剧本「确认点」。
- Playbook 状态机字段：`current_step`、`task_status` 存于 PlatformContext / 平台工单。

---

## 2. 剧本总表（6 个）

| ID | 名称 | 已实现 | 服务调用链（逻辑序） | 典型触发 | 确认点 |
|----|------|:------:|----------------------|----------|--------|
| `shift_patrol` | 开班巡检 | N | patrol → triage? | 巡检员开班 | triage 可选 |
| `trace_only` | 批次查询 | Y | trace | 查批次 / 显式 API | 无 |
| `investigate` | 深度分析 | Y | triage? → trace → rca | 工程师一键分析 | **trace → rca 前** 确认 |
| `close_loop` | 质量闭环 | Y | rca → hitl → report_8d → qms写回 | 根因确认后 | **8D 前** rca done + HITL |
| `coating_incident` | 涂布异常 | N | quality-pred → process → safety ∥ rca → report_8d | SCADA/SPC 告警 | 改参经 Safety HITL |
| `pm_alert` | 设备预警 | N | equipment-health → process → safety | EAM/振动告警 | 停线/改参经 Safety |

**快捷入口**：`task_type=rca` 单点调用 quality-rca-agent（无前置 trace），见 §3.4；不计入上表 6 剧本。

> **并行**：`coating_incident` 中工艺链与 RCA→8D 链可并行，由 Router fork 子任务。

## 3. 各剧本详述

### 3.1 `shift_patrol` — 开班巡检

| 步骤 | Agent | capability | 写入 Context |
|------|-------|------------|--------------|
| 1 | patrol-agent | patrol_summary | `shift_summary` |
| 2? | triage-agent | defect_classification | `triage_result`, `severity` |

**结束条件**：patrol 返回摘要；若 `severity >= medium` 可建议 Router 提示是否启动 `investigate`。

---

### 3.2 `trace_only` — 批次查询

| 步骤 | Agent | capability | 写入 Context |
|------|-------|------------|--------------|
| 1 | trace-agent | batch_trace | `prior_evidence`, `prior_tool_calls` |

---

### 3.3 `investigate` — 深度分析

| 步骤 | Agent | capability | 写入 Context |
|------|-------|------------|--------------|
| 1? | triage-agent | defect_classification | `defect_type`, `severity`, `triage_result` |
| 2 | trace-agent | batch_trace | `prior_evidence` |
| — | **确认** | — | 用户/UI 或 API `confirm_rca: true` |
| 3 | quality-rca-agent | root_cause_analysis | `rca.*` |

---

### 3.4 `rca` — 单点根因（快捷，非独立 Playbook）

| 步骤 | Agent | capability | 说明 |
|------|-------|------------|------|
| 1 | quality-rca-agent | root_cause_analysis | `task_type=rca` 或 API 直调；可带 `prior_evidence` |

---

### 3.5 `close_loop` — 质量闭环

| 步骤 | Agent | capability | 说明 |
|------|-------|------------|------|
| 1 | quality-rca-agent | root_cause_analysis | 直至 `rca.status=done` |
| 2 | — | HITL | L1/L2 按置信度与严重度 |
| 3 | report-reporter-agent | report_8d | **定稿** CAPA；写 `report_8d.report_md` |
| 4 | qms MCP | upload_quality_report | Harness 代调；非 Agent |

**RCA vs 8D 分工**：

| 组件 | 产出 | 阶段 |
|------|------|------|
| RCA **Reporter** 节点 | 8D **草稿**（`artifacts`） | 分析过程中 |
| **report-reporter-agent** | 定稿报告 + QMS 写回 | HITL 通过后 |

---

### 3.6 `coating_incident` — 涂布异常

| 分支 | 步骤 | Agent | 说明 |
|------|------|-------|------|
| A 工艺 | 1 | quality-prediction-agent | SPC/面密度告警 |
| A | 2 | process-optimization-agent | 粘度/温度取证与建议 |
| A | 3 | safety-agent | 改参 HITL → `plc` MCP |
| B 质量 | 1 | quality-rca-agent | 并行根因链 |
| B | 2 | report-reporter-agent | 证据链落 QMS |

详见 [260630_架构A2A_MCP.md](./260630_架构A2A_MCP.md) 图 5 时序。

---

### 3.7 `pm_alert` — 设备预警

| 步骤 | Agent | capability | 说明 |
|------|-------|------------|------|
| 1 | equipment-health-agent | predictive_maintenance | 振动/电流频谱 |
| 2 | process-optimization-agent | parameter_recommendation | 工艺联动建议 |
| 3 | safety-agent | parameter_write_approval | 停线/改参门闩 |

## 4. API 入参

```json
{
  "session_id": "sess-001",
  "message": "分析批次 B20260623001",
  "playbook": "investigate",
  "factory_id": "FD-01",
  "batch_id": "B20260623001",
  "task_type": "assistant",
  "confirm_rca": false
}
```

| 字段 | 说明 |
|------|------|
| `playbook` | 上表 ID；缺省时 Client 按 `task_type` + 规则推断 |
| `confirm_rca` | `investigate` 中 trace 完成后是否直接进 RCA |
| `task_type` | `rca` 时等价 `playbook=rca` |

---

## 5. 状态机（平台工单）

```
received → running_step_N → input_required → running_step_N+1 → done
                         ↘ failed / cancelled
```

| `current_step` 示例 | 对应 Agent |
|---------------------|------------|
| `patrol` | patrol-agent |
| `trace` | trace-agent |
| `rca` | quality-rca-agent |
| `hitl` | RCA INPUT_REQUIRED |
| `report_8d` | report-reporter-agent |
