## 二、项目概览

### 锂电质量闭环智能体平台

**技术栈**：Python · FastAPI · LangGraph · Deep Agents · MCP · A2A · Redis · Neo4j · Milvus · LLM（DeepSeek / 兼容 OpenAI SDK)· Docker

#### 做什么

锂电池制造有 10+ 道工序，数据散落在 MES、SCADA、ERP、LIMS 四套系统里。出现质量异常时——比如一批电芯容量偏低——工程师要手工跨系统取证，一次排查 4 到 8 小时，而且结论靠经验、过程不可复现。这不是加一个 Chatbot 能解决的。

我做的是一个 **Agent 原生平台**，让 3 个 Agent 自动协作完成整条链路：用户说「帮我查这批电池怎么回事」→ Planner 理解意图选剧本 → Trace Worker 跨 MCP 取证 → RCA Agent（LangGraph + FMEA 规则引擎）锁定根因 → Reporter Agent（Deep Agent）生成 8D 报告 → 自动写回 QMS 系统。整条链路带 trace_id 全审计，人只在置信度不够时介入签核。

#### 怎么跑的

```
用户 ──→ Gateway (8010) ──→ Planner (8011) ──→ Orchestrator (8020)
                                                    │
                              ┌──────────────────────┼──────────────────────┐
                              ▼                      ▼                      ▼
                         Trace Worker           RCA Agent            Reporter Agent
                           (8002)                (8003)                 (8004)
                          MCP 取证         LangGraph 4 节点推理+      Deep Agent 8D
                              │               HITL 中断点       ┌──┘
                        mes · scada       planner executor  d5_capa_planner
                                              │  reflector     (唯一子Agent)
                                              │  (interrupt())  D4/D6 预格式化
                                              │  reporter      主Agent拼装
                                              │
                            mes(8101)  scada(8102)  erp(8103)  lims(8104)  qms(8105)
                            knowledge(8106)  wms(8107)  plc(8108)  eam(8109)
                                       MCP Servers × 9
```

- **控制面**：Gateway（入口 + SSE 推送）→ Planner（ReAct + rule 双模，LLM 挂了自动降级）→ Orchestrator（星型调度，Agent 间不直连）
- **业务层**：Trace Worker（纯 MCP 取证，不用 LLM）、RCA Agent（LangGraph 有环图 + FMEA 规则引擎）、Reporter Agent（Deep Agent 8D 定稿 + QMS 写回）
- **基础设施**：Redis（短期记忆）+ Neo4j（长期因果图）+ Milvus（向量索引）+ 9 个 MCP Server（含 knowledge 混合检索），Working Memory 层预留 PostgreSQL 接口

## 三、核心技术点

### 1. Agent 间通信 — 为什么选 A2A 而不是 gRPC

所有 Agent 通过 `python_a2a` 协议通信：`AsyncA2AServer` 基类统一接入，`mount_a2a()` 在 FastAPI 上挂载 `/.well-known/agent.json`（AgentCard 自发现）、`/tasks/send`（任务委派）、`/tasks/resume`（HITL 恢复）三个标准端点。A2AClient 封装了 `Task.from_dict()` 和 Message/TaskState 序列化，auth 层支持 JWT + service key 双模式。

选 A2A 的核心原因就一个：**gRPC 只有成功和失败，A2A 多了个 `INPUT_REQUIRED`**——这个状态对应 LangGraph 的 `interrupt()` 挂起等人签核。制造场景里质量经理签字是刚需，gRPC 表达不了「任务没失败，但在等一个人」。

A2A 的缺点也直接：JSON-RPC over HTTP 没有内置流控，大文件不适合嵌在 Task artifacts 里，`tasks/resume` 的恢复语义和 LangGraph checkpoint thread_id 之间有一层映射 gap——这些我都做了显式处理。

> **追问预测 — A2A 的局限**：JSON-RPC over HTTP 无流控/背压；大文件（8D PDF）不适合嵌入 artifacts；resume 语义在 A2A 和 LangGraph `interrupt()` 之间有一层 gap（status vs checkpoint thread_id）。生产化可考虑 AGNTCY SLIM 或 WebSocket 替代 SSE。

---

### 2. RCA Agent — LangGraph 推理 + 规则引擎，而不是纯 LLM

RCA Agent 做根因分析，图结构是 Planner → Executor → Reflector → Reporter 四个节点，Reflector 内嵌 `interrupt()` 实现 HITL 挂起。允许 Reflector 根据证据充足度触发 DEEPEN（深挖）/ CORRELATE（关联）/ REPLAN（重规划）三种策略回流，最多三轮。

**为什么 LangGraph 而不是 CrewAI/AutoGen？** RCA 的推理拓扑是固定的——不是开放式对话，不需要多 Agent 协商。我需要的是：每条推理边可追溯（审计）、FMEA 因果树可注入（领域知识）、推理中段可挂起等人签字（HITL）。LangGraph 的显式 StateGraph + conditional edges + `interrupt()` 原生支持这三件事，CrewAI 的对话式多 Agent 协作做不到。

**为什么置信度不用 LLM 自评？** LLM 说「我有 85% 把握」本质上是一个文字接龙概率，同一条输入问两次可能不一样。制造场景要过 IATF 16949 审计——置信度必须可复现。我的方案：Reflector 用 FMEA 规则引擎算 `evidence_coverage × causal_chain_completeness`，低于 0.7 强制挂 HITL。目前 FMEA CSV 只有容量衰减一个缺陷类型——这是已知限制，未命中时 Reflector 走 DEGRADE 模式（纯 LLM + 标记低置信度 + 强制签核）。

**HITL 断点怎么和 A2A 配合？** Executor 取证 → Reflector 判定低置信度 → `interrupt()` 挂起 → A2A 返回 `INPUT_REQUIRED` → 质量经理签核 → Gateway 调 `tasks/resume` → `Command(resume=feedback)` 从 checkpoint 恢复继续跑。两套状态机之间的 thread_id ↔ session_id 映射由 `AsyncA2AServer._thread_map` 维护。

> **追问预测 — FMEA 覆盖率不足**：当前 Golden Case 库已覆盖 15 例（15条，含12种缺陷类型、9道工序）。Reflector 的 DEGRADE 模式在 FMEA 无命中时降级为纯 LLM + 低置信度标记，强制 HITL 签核。长期方案走 Milvus 向量 + Neo4j 因果图**混合检索**（`hybrid_search_golden_case`），即使 defect_type 不在图库里也能通过语义相似捞出参考案例。  
> **追问预测 — RCA 超时/故障**：Orchestrator 侧兜底——从 trace evidence + FMEA 构造初判根因，标记低置信度，强制走人工签核，不阻塞管线。

### 3. Reporter Agent — 不是所有子任务都需要 Agent

Reporter 负责把 RCA 产出的根因和证据组装成完整 8D 报告（D1-D8），写回 QMS。最初设计按 D4/D5/D6 各分一个子 Agent，跑起来发现 25 次 LLM 调用里 D4（贴根因）和 D6（编号列表）各消耗了 6-7 次——它们本质上就是字符串格式化，根本不需要 LLM 判断。

**改了三版**：v1（3 子 Agent）25 次调用 94s → v2（换成 tool 函数）直接卡死（tool 返回字符串，LLM 不知道「已完成」，反复调用）→ v3（**D4/D6 预格式化注入 + 只有 D5 一个子 Agent**）**9 次调用 42s**。

核心认知：**子 Agent 的粒度应该由「搜索歧义」决定，不是 8D 章节数**。D5（纠正措施）需要查 SOP、匹配历史 Golden Case——有搜索不确定性，需要 LLM 判断——所以保留为唯一子 Agent `d5_capa_planner`。检索层已从 JSON 精确匹配升级为 **Milvus 向量 + Neo4j 知识图谱混合搜索**（`hybrid_search_golden_case`）：向量找语义相似的，图找因果相连的，RRF 融合排序后作为 few-shot 注入子 Agent。D4/D6 在 Python 里拼好字符串，用 `[D4_PREFORMATTED]` 标签注入 user message，主 Agent 收到的就是成品 Markdown，直接拼装，零 tool 调用、零循环风险。

D4 根因锁定的三层防御：(1) Python 字符串拼接，不经过 LLM；(2) system prompt 标注「禁止调 tool 获取，原样使用」；(3) 离线 `rule_judge_reporter_d4_locked()` 断言检查。

LLM 不可用或 `deepagents` 没装 → 自动回退模板模式。`recursion_limit=25` 是安全天花板——v3 正常路径 ~9 步结束，只有死循环才触发。

> **追问预测 — 为什么砍到 1 个子 Agent**：d4/evidence 只是「调 tool 取数据→格式化」，确定性操作不需要 LLM。把它们从子 Agent 降级为预格式化块后，LLM 调用从 25 降到 9。粒度由搜索歧义决定，不是章节数。  
> **追问预测 — recursion_limit=25 是拍脑袋吗**：LangGraph 默认 25。v1 实测恰好 25 步用完，v3 优化后自然 ~9 步结束。25 是天花板——正常路径不触发，死循环兜底。

### 4. Planner Agent — 能 ReAct 就 ReAct，不行就 rule

Planner 只有一件事：把用户的自然语言转成「该跑哪个 playbook + 什么参数」。ReAct 模式下 LLM + 3 个 tool（`list_playbooks` / `get_capability_card` / `submit_plan`），LLM 不可用时自动降级为正则 + 关键词匹配的 rule 引擎。最多 8 轮 ReAct 循环，实际正常意图 2-3 轮就结束了。规划工具是进程内的——Planner 不需要访问制造数据，只读 AgentCard 元数据。

**为什么不用 LangGraph？** Planner 的推理是标准的 ReAct 链（Thought→Action→Observation→Final），不是有环图。LangGraph 的 StateGraph + checkpoint 开销对它是过度设计。未来如果需要多步 DAG 编排（含并行分支），再考虑迁移。

> **追问预测 — 降级安全**：LLM 不可用（没配 key 或 API 挂了）→ 自动切 rule 引擎。rule 模式覆盖所有注册 playbook，有 batch_id 时置信度 0.75，没有时 0.6——足够 Orchestrator 做决策。

### 5. 星型拓扑 + PlatformContext 黑板

Agent 之间不直连。所有跨 Agent 调度经 Orchestrator 统一委派，跨 Agent 状态走 PlatformContext 黑板（`prior_evidence`、`rca.root_cause`、`report_8d.capa_id` 等），每次调用带 `trace_id`。

**为什么这么设计？** 三个原因：(1) 解耦——换 Reporter 实现只改 Orchestrator 一个函数，RCA 不需要知道下游是谁；(2) 故障隔离——Reporter 挂掉不影响 RCA 完成根因分析；(3) 审计——所有 A2A 调用经过单一入口，trace_id 天然贯穿全链路。代价是一跳额外的 HTTP（~10ms）和一层 Context 序列化，但在这个低频、长周期、有人工断点的场景里，延迟不是瓶颈，可追溯性才是。

Context 分层：PlatformContext 存的是下游 Agent **需要**的数据（root_cause、capa_id），Agent 自己的 STM/Working/LTM 是**内部记忆**，不对外暴露。两层的 key 没有交集，避免了「一个巨无霸 Context 什么都在里面」的混乱。

> **追问预测 — Orchestrator 单点**：当前是单实例。Orchestrator 本身无状态（Context 在 Redis），可水平扩展；任务状态机幂等（playbook + session_id 去重）；生产化需补 Saga 补偿逻辑——目前降级路径已经覆盖了关键故障点。

---

### 6. MCP 工具层 + 安全基础设施

9 个制造域（MES/SCADA/ERP/LIMS/QMS/Knowledge/WMS/PLC/EAM）各一个 MCP Server，FastMCP SSE transport，独立进程。Agent 通过 ToolRegistry 统一调用，每次经过 RBAC 校验 + trace_id 审计 + 指数退避重试。

**设计原则**：MCP 层不实现业务逻辑——一个 Tool 就是一个参数化数据查询。Harness（权限/审计/重试/熔断/会话/事件）拆成独立的 `harness-core` package，不绑定任何 Agent。

数据安全上做了两层：(1) `mcp_tool_matrix.py` 定义每个 Agent 的 Tool 白名单——Reporter 调不了 MES，RCA 调不了 QMS；(2) `plc.emergency_stop` 和 `plc.write_setpoint` 通过 `exclusive_agent` 字段独占绑定 safety-agent——非授权 Agent 在 PermissionChecker 层直接拦截，不会到达 MCP Server。

> **追问预测 — MCP vs REST**：MCP 的价值是 Agent 侧自发现——`list_tools()` 拿 tool schema，不需要硬编码 API endpoint。但 SSE transport 在 Windows 上有连接稳定性问题（实测遇到过 TaskGroup 异常），生产化建议用 Streamable HTTP。

---

### 7. LLM 模型路由 — 不是每个 Agent 配一个模型

按 `DataSensitivity`（LOW/MEDIUM/HIGH）× `TaskDifficulty`（TRIVIAL/SIMPLE/MODERATE/COMPLEX）做两层路由，不是按 Agent 分——那会导致配置爆炸。

路由逻辑：sensitivity 决定数据去哪儿（external API vs local vLLM），difficulty 决定用哪个模型（flash vs primary）。Reflector 处理工艺参数摘要 → MEDIUM → local vLLM 优先；Planner/Reporter 处理缺陷描述 → LOW → external DeepSeek。HIGH 级别当前没有任何节点标记——MES/SCADA 原始数据只在 Executor 做规则匹配，不进 LLM prompt。

10 个敏感 key（`recipe_params`、`equipment_id`、`lot_number` 等）在送外部 API 前递归脱敏为 `[REDACTED]`。每次 LLM 调用的路由决策（model/endpoint/sensitivity/difficulty/caller）写入审计日志。Local vLLM 当前未部署，MEDIUM 调用走 external 降级 + 审计告警——数据安全上不够完美，但比不做分级直接裸送强。

> **追问预测 — HIGH 级数据走 external**：当前无节点标记 HIGH。原始数据不进 LLM。如果未来需要（如 PLC 自动调参），`_resolve_route()` 直接抛异常，不降级。  
> **追问预测 — local/external 能力差距**：local 部署 qwen2.5-72b，能力弱于 deepseek。但 Reflector 的 LLM 路径是 FMEA 规则引擎的补充——规则命中时完全不调 LLM，只在多因耦合时才调——此时宁可保守（低置信度→HITL），不冒险送外部。

---

### 8. 评测体系

两层评测：规则 Judge 做确定性断言（根因是否为空、原文是否被改写、证据链是否完整）——0 token 成本，每次提交可跑。LLM-as-Judge 做语义一致性审计——配了 API key 才启用，不配就跳过不阻塞。

Golden Set 25 例（RCA 10 + Reporter 15），`run_eval.py --all` 一键跑完。5 项 Prompt 评估目标写入 `registry.yaml`（d4_locked =1.0 / plan_validity ≥0.90 / playbook_accuracy ≥0.90 / fmea_hit_rate ≥0.80），改 Prompt 前跑一次回归对比。

## 四、面试话术

### 30 秒电梯

> 我做了个锂电质量闭环平台——3 个 Agent 自动协作，把「用户说有问题」到「完整 8D 报告写回 QMS」全自动化了。核心亮点三个：第一，RCA Agent 的根因置信度是 FMEA 规则引擎算的，不是 LLM 自评——同一个输入永远同一个分数，质量经理敢签字；第二，Reporter Agent 的 D4 根因是 Python 字符串拼的，不经过 LLM，从代码层面锁死不可改写；第三，LLM 调用做了两层路由——工艺参数走本地模型优先，敏感数据不出厂。15 个容器 Docker Compose 一键启动，全链路跑通 ~55 秒（含 42s Deep Agent 8D 生成），人工排查从 4-8 小时缩短 260-520 倍。

### 2 分钟 STAR

**S**：锂电质量异常排查，工程师要手工跨 MES/SCADA/ERP/LIMS 四个系统取证，一次 4 到 8 小时。已有的 Chatbot 方案只能聊天，做不到可复现判定和可审计输出。

**T**：我负责整个质量闭环链路的设计和实现——从用户输入到 8D 报告输出。

**A**：平台层用 A2A 协议做 Agent 间通信，Orchestrator 星型调度。RCA Agent 用 LangGraph 四节点推理图加 FMEA 规则引擎——置信度规则算不是 LLM 自评，低置信度强制挂起等人签核。Reporter Agent 用 Deep Agent 子 Agent 委派——D4 根因预格式化注入，从代码层面防改写。整条链路 trace_id 贯穿，每次 MCP 调用经 RBAC + 指数退避重试。

**R**：15 容器一键启动。全链路（含 Deep Agent 8D 生成）从人工 4-8 小时缩短到 ~55 秒（效率提升 260-520×），其中 Reporter 从最初 25 次 LLM 调用优化到 9 次（↓64%），耗时 94 秒压到 42 秒。14 个 MCP 工具已实现（含 knowledge 混合检索），25 例 Golden Set 评测就绪。模型路由、RBAC、安全门闩、3 条降级路径全部上线。

### 最难的技术决策

> 最难的是决定 **RCA 和 Reporter 不直连**，中间插一层 Orchestrator 加 PlatformContext 黑板。
>
> 直连方案少写代码、少一跳延迟。但直连意味着 RCA 要知道 Reporter 的接口——换一个 Reporter 实现就要改 RCA，加一个审核节点要改两边。这在制造业是致命的——IATF 16949 要求每个环节可独立验证，不能因为「改了 8D 模板」就重新验证根因分析模块。
>
> 引入 Orchestrator 中转的代价是一跳 HTTP（~10ms）和多一层 Context 序列化。但换回来的是：RCA 和 Reporter 独立演进，各自的模型、权限、部署周期互不感知；所有跨 Agent 状态在 PlatformContext 里完整可审计；未来插一个新 Agent 只改 Orchestrator 一个文件。
>
> 这个取舍背后是个更底层的问题——**制造场景的变更管理需求远比互联网场景严格**。互联网可以「先上线再说」，制造业一旦过了验证就不能随便改。

### 如果重新做，会改什么

> 第一，PlatformContext 当前是 Pydantic 强类型字段——每加一个 Agent 新产出就要改 Schema。应该用 artifacts blob + JSON Schema 验证，让下游按需取用而不是上游预定义。
>
> 第二，Orchestrator 的 playbook 是 if-else 硬编码，`_run_playbook()` 里 500 行——应该抽成 DSL 或 YAML 配置，让工艺工程师也能编排剧本。目前已解决：`PlaybookEngine` 从 `config/playbooks.yaml` 驱动执行，新增剧本只需加 YAML + AgentCard。
>
> 第三，MCP Server 的 stub 数据太理想化——应该至少用 Faker 生成边界数据，测得出 failover 才叫「韧性就绪」。
>
> 第四，评测闭环还不够——Golden Set 25 例偏少，RCA 端到端评测（从输入到根因的 F1/nDCG）还没跑起来。

**平台架构** — Agent 间用 python_a2a 协议通信，选 A2A 而不是 gRPC 是因为它的 `INPUT_REQUIRED` 状态直接对应 LangGraph `interrupt()` 的人等签核场景——gRPC 只有成功和失败，表达不了「任务没失败，但在等一个人」。Orchestrator 星型调度，Agent 不直连——跨 Agent 状态走 PlatformContext 黑板，每次调用带 trace_id 全链路可审计。Gateway 统一入口 + SSE 推送 → Planner（ReAct + rule 双模，LLM 不可用自动降级）→ Orchestrator 按 Playbook 寻址委派。9 个制造域（MES/SCADA/ERP/LIMS/QMS/Knowledge/WMS/PLC/EAM）各一个 MCP Server，Agent 通过 ToolRegistry 统一调用，每次经 RBAC 校验 + 指数退避重试。`plc.emergency_stop` 和 `plc.write_setpoint` 通过 `exclusive_agent` 字段独占绑定 safety-agent，非授权调用在 PermissionChecker 层拦截。

**RCA Agent** — LangGraph 四节点有环图（Planner→Executor→Reflector→Reporter），Reflector 内嵌 interrupt() 实现 HITL 挂起，Reflector 支持 DEEPEN/CORRELATE/REPLAN 三种策略回流，最多 3 轮。根因置信度由 FMEA 规则引擎计算（`evidence_coverage × causal_chain_completeness`），不是 LLM 自评——同一个输入永远同一个分数，质量经理敢签字。低置信度时 `interrupt()` 挂起，A2A 返回 `INPUT_REQUIRED`，人签完从 LangGraph checkpoint 恢复继续跑。RCA 超时或故障不阻塞管线——Orchestrator 从 trace evidence 构造初判根因，标记低置信度，强制 HITL。

**Reporter Agent** — 基于 `deepagents.create_deep_agent()`，主 Agent + 1 个子 Agent（`d5_capa_planner`）。D4 根因和 D6 证据在 user message 中预格式化注入（Python 字符串拼接，不走 LLM），D5 纠正措施委派给子 Agent 查 SOP/Golden Case 后生成。架构从 3 子 Agent 25 次 LLM 调用优化到 1 子 Agent **9 次（↓64%）**，94s→**42s（↓55%）**。核心经验：子 Agent 粒度由「搜索歧义」决定，确定性操作不需要 LLM。

**模型路由与安全** — 按 `DataSensitivity`（LOW/MEDIUM/HIGH）× `TaskDifficulty`（TRIVIAL→COMPLEX）两层路由，不是按 Agent 数量膨胀配置。Reflector 处理工艺参数摘要标记 MEDIUM 优先走本地 vLLM，Planner/Reporter 标记 LOW 走外部 DeepSeek。10 个工艺机密 key 递归脱敏后再送外部 API。每次 LLM 路由决策写入审计日志。

**工程化** — 熔断器（3 次失败→OPEN→5% 探针概率恢复）+ 指数退避重试 + 3 条降级路径。评测：规则 Judge（确定性断言，0 token）+ LLM-as-Judge（可选）。Golden Set 25 例。Prompt 版本化管理，改前跑回归。

## 六、面试速答卡

**Q：为什么不用 LangChain 全套？**
LangGraph 的 `interrupt()` 做 HITL 断点恢复，Deep Agents 做子任务委派——LangChain AgentExecutor 两个都不支持。

**Q：A2A vs gRPC？**
gRPC 只有成功/失败两个状态。A2A 多了 `INPUT_REQUIRED`——对应人等签核——制造场景的刚需。

**Q：Agent 为什么不直连？**
解耦。换 Reporter 只改 Orchestrator 一个函数。PlatformContext 黑板让每一跳可审计。新 Agent 插进来只动 Orchestrator。

**Q：FMEA 怎么维护？**
CSV 是 Source of Truth，启动时加载。生产化走 **ETL 管道**（`scripts/etl_golden_cases.py`）：CSV → embedding → Milvus 向量索引 + Neo4j 图节点/关系同步。工艺变更同步更新因果树，带版本号。RCA Agent 通过 `knowledge.hybrid_search_golden_case` MCP Tool 同时查询两种索引。

**Q：怎么保证 LLM 不改写根因？**
三层：RCA 置信度是 FMEA 规则算的（不是 LLM 自评）；Reporter 的 D4 是 Python 字符串拼的（不经过 LLM）；离线 rule_judge 跑断言兜底。

**Q：RCA 超时或挂了？**
不阻塞。Orchestrator 从 trace evidence + FMEA 构造初判根因，标记低置信度，强制走人工签核。

**Q：多 Agent 模型怎么选型？**
不是每个 Agent 一个模型——配置爆炸。按数据敏感度 × 任务难度两层路由：工艺参数走本地优先，缺陷描述走外部 API。敏感字段递归脱敏。

**Q：并发和部署？**
MCP 取证 `asyncio.gather` 并行。Docker Compose 一键启动 15 容器。session_id 隔离多用户。RCA Agent 通过 Junction 链接姊妹仓。
